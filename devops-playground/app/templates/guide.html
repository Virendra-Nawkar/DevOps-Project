2 Load Generator — Trigger Autoscaling

Start CPU stress workers inside the pod. When CPU crosses 50%, Kubernetes HPA automatically creates new pods. Stop the load and pods scale back down after ~5 minutes.

POST /load/start
{"workers":4}
→
CPU spikes
to 80-100%
→
HPA fires
(CPU > 50%)
→
1 → 3 → 5
pods created
curl — start load
curl -X POST http://52.173.127.47:80/load/start \
  -H "Content-Type: application/json" \
  -d '{"workers": 4}'
curl — stop load
curl -X POST http://52.173.127.47:80/load/stop
Python — full autoscaling trigger
import requests, time

BASE = "http://52.173.127.47:80"

# Start load with 4 workers
r = requests.post(f"{BASE}/load/start", json={"workers": 4})
print("Load started:", r.json())

# Watch CPU every 5 seconds
for _ in range(6):
    info = requests.get(f"{BASE}/info").json()
    print(f"CPU: {info['cpu_percent']}%  Workers: {info['active_load_workers']}")
    time.sleep(5)

# Stop load
requests.post(f"{BASE}/load/stop")
print("Load stopped. Pods will scale down in ~5 min.")

Watch scaling in Kubernetes: kubectl get hpa -n devops -w and kubectl get pods -n devops -w

3 Chaos Engineering

Break things on purpose to test how Kubernetes responds. This is how Netflix, Amazon, and Google test their systems are resilient.

ActionWhat you sendWhat happensKubernetes reaction
Kill Liveness POST /chaos/unhealthy /health returns 500 Pod restarted after 3 failures (~30s)
Restore Health POST /chaos/healthy /health returns 200 Pod stays running normally
Kill Readiness POST /chaos/not-ready /ready returns 503 Pod removed from load balancer, no traffic
Restore Ready POST /chaos/ready /ready returns 200 Pod added back to load balancer
Inject Latency POST /chaos/delay {"ms":500} Every response delayed 500ms SLA breach visible in Grafana
Trigger Error GET /chaos/error Returns HTTP 500 Error counter increments, alert may fire
curl — all chaos commands
# Break liveness probe (pod will restart in ~30s)
curl -X POST http://52.173.127.47:80/chaos/unhealthy

# Restore liveness
curl -X POST http://52.173.127.47:80/chaos/healthy

# Remove pod from load balancer
curl -X POST http://52.173.127.47:80/chaos/not-ready

# Add it back
curl -X POST http://52.173.127.47:80/chaos/ready

# Inject 500ms latency
curl -X POST http://52.173.127.47:80/chaos/delay \
  -H "Content-Type: application/json" \
  -d '{"ms": 500}'

# Remove latency
curl -X POST http://52.173.127.47:80/chaos/delay \
  -H "Content-Type: application/json" \
  -d '{"ms": 0}'

# Trigger a 500 error
curl http://52.173.127.47:80/chaos/error
Python — chaos test sequence
import requests, time

BASE = "http://52.173.127.47:80"

# Test 1: break and auto-restore liveness
requests.post(f"{BASE}/chaos/unhealthy")
print("Liveness broken — Kubernetes will restart pod in ~30s")
time.sleep(35)
requests.post(f"{BASE}/chaos/healthy")
print("Liveness restored")

# Test 2: latency injection
requests.post(f"{BASE}/chaos/delay", json={"ms": 500})
print("Added 500ms delay")
time.sleep(10)
requests.post(f"{BASE}/chaos/delay", json={"ms": 0})
print("Delay removed")

5 Blue / Green Deployment

There are two versions of the app running simultaneously. A router on port :85 splits traffic between them. Drag the slider on the dashboard — or use the API — to shift 0%→100% of traffic to either version. This is how companies do zero-downtime deployments.

User request

→
🔵 Blue

curl — check current traffic split
curl http://52.173.127.47:85/router/weight
curl — shift traffic
# Send 50% to blue, 50% to green
curl -X POST http://52.173.127.47:85/router/weight \
  -H "Content-Type: application/json" \
  -d '{"blue": 50, "message": "Canary: 50/50 split"}'

# All traffic to green (new version)
curl -X POST http://52.173.127.47:85/router/weight \
  -H "Content-Type: application/json" \
  -d '{"blue": 0, "message": "Full cutover to green v2"}'

# Rollback to blue
curl -X POST http://52.173.127.47:85/router/weight \
  -H "Content-Type: application/json" \
  -d '{"blue": 100, "message": "Rollback to blue v1"}'
Python — automated canary rollout
import requests, time

ROUTER = "http://52.173.127.47:85"

steps = [
    (90, "Canary: 10% to green"),
    (70, "Canary: 30% to green"),
    (50, "Canary: 50/50 split"),
    (20, "Canary: 80% to green"),
    (0,  "Full cutover to green v2"),
]

for blue_pct, msg in steps:
    r = requests.post(f"{ROUTER}/router/weight",
                      json={"blue": blue_pct, "message": msg})
    print(f"✓ {msg}", r.json())
    time.sleep(30)  # wait and observe error rate before next step

print("Deployment complete — 100% traffic on green v2")
6 Live Metrics Stream (SSE)

/stream uses Server-Sent Events — the server pushes a JSON update to you every 2 seconds without you having to poll. The dashboard uses this to update CPU/memory charts in real time. You can consume it from curl or Python too.

curl — watch live stream in terminal (Ctrl+C to stop)
curl -N http://52.173.127.47:80/stream

# Output (every 2 seconds):
# data: {"cpu": 12.3, "memory": 45.1, "requests": 84, "workers": 0, ...}
# data: {"cpu": 14.7, "memory": 45.2, "requests": 85, "workers": 0, ...}
Python — consume SSE stream
import requests, json

url = "http://52.173.127.47:80/stream"
resp = requests.get(url, stream=True)

print("Listening to live stream... (Ctrl+C to stop)")
for line in resp.iter_lines():
    if line and line.startswith(b"data:"):
        data = json.loads(line[5:])
        print(
            f"CPU:{data['cpu']:5.1f}%  "
            f"MEM:{data['memory']:5.1f}%  "
            f"REQ:{data['requests']:6d}  "
            f"WORKERS:{data['workers']}"
        )

8 Kubernetes Commands (for operators)

These commands work on the VM when K3s is running. Run them in a separate terminal while you trigger load from the browser.

Watch pods scale in real time
kubectl get pods -n devops -w
Watch HPA (autoscaler) decisions
kubectl get hpa devops-playground -n devops -w
See CPU + memory per pod
kubectl top pods -n devops
See scaling events
kubectl get events -n devops --sort-by='.lastTimestamp' | grep -i scal
Stream logs from all pods
kubectl logs -l app=devops-playground -n devops -f
All useful commands at once
# Scale manually (bypasses HPA)
kubectl scale deployment devops-playground --replicas=5 -n devops

# Get shell inside a running pod
kubectl exec -it -n devops \
  $(kubectl get pod -n devops -l app=devops-playground -o name | head -1) -- bash

# Rolling update to new image
kubectl set image deployment/devops-playground \
  app=devops-playground:latest -n devops

# Rollback
kubectl rollout undo deployment/devops-playground -n devops

# Delete a pod (K8s recreates it immediately)
kubectl delete pod -n devops \
  $(kubectl get pod -n devops -o name | head -1)
